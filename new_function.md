# 機能追加案の評価・優先度一覧

現状の時間割Generatorは、**科目マスタ（`subject`）を CLI で PDF から投入**し、**Web UI 上でコマ単位に履修登録（`enrollment`）**する二層構成です。以下では各案を **実現可能性**（技術的難易度・既存資産との親和性）と **有用性**（利用頻度・課題解決度）の観点で評価し、**実装優先度が高い順**に並べています。

---

## 評価の凡例

| 記号 | 実現可能性 | 有用性 |
|------|------------|--------|
| ◎ | 既存コードの延長で短期実装可能 | 日常的に効く・汎用性が高い |
| ○ | 中程度の工数・一部設計が必要 | 明確なメリットがある |
| △ | 仕様整理・品質担保に工数がかかる | ニッチまたは効果が限定的 |
| × | 現状アーキテクチャと相性が悪い／効果が薄い | 優先すべきでない |

---

## 優先度順：機能一覧

### 1. PDF アップロード＋読み取りページ指定（ユーザー案 ①）

| 観点 | 評価 | コメント |
|------|------|----------|
| 実現可能性 | ◎ | `scripts/create_data_pdf.py` の `process_timetable_pdf` を FastAPI の `UploadFile` 経由で呼び出す形に移せる。ページ指定は `page_index` の UI 化で足りる。 |
| 有用性 | ◎ | `scripts/table.pdf` の事前配置と `uv run python -m scripts.create_data_pdf` を不要にし、**他年度・他大学への展開の入口**になる。 |

**補足・リスク**

- 現行パーサは **特定 PDF の列レイアウト**（`row[4]` の「全」「実」、`row[1]` が数字など）に強く依存している。アップロード後も **取り込み失敗・誤パース**は起こりうる。
- README にある **再実行時の重複登録**は Web 化後も残る。優先度を上げるなら **「取り込み前に既存 subject をクリアする」** または **授業コードで upsert** をセットで設計すべき。
- 月〜金とページの対応（現状 2〜6 ページ目）をユーザーが指定できる UI（曜日ごとにページ番号、または「開始ページから連番」）があると案①の意図に沿う。

**優先度が高い理由**: データ投入が Web 完結になり、アプリの「汎用ツール化」の前提条件になる。

### 2. 科目名・授業コード検索

| 観点 | 評価 | コメント |
|------|------|----------|
| 実現可能性 | ◎ | `get_all_subjects` が既にある。`LIKE` 検索の API と検索 UI を追加する程度。 |
| 有用性 | ◎ | コマを開かずに科目を探せる。科目数が多い学期ほど効果が大きい。 |

**補足**

- 検索結果から **該当コマへスクロール** または **ワンクリックで履修登録** までつなげると体験が一段上がる。

### 3. フィルター設定・回生の永続化（localStorage）

| 観点 | 評価 | コメント |
|------|------|----------|
| 実現可能性 | ◎ | フロントのみ。DB 変更不要。 |
| 有用性 | ○ | 毎回 1 回生・分野を選び直す手間を削減。ログイン機能がない現状でも合理的。 |

### 4. 履修データのエクスポート／インポート（JSON）

| 観点 | 評価 | コメント |
|------|------|----------|
| 実現可能性 | ◎ | `get_enrollments` と `add_enrollment` の組み合わせで実装可能。 |
| 有用性 | ○ | 端末変更・DB リセット・複数案の比較時にバックアップとして有効。 |

**補足**

- 画像 PDF 出力より先に JSON の方が工数対効果が良い。画像化は後続でもよい。

---

### 5. PDF 取り込み結果のプレビュー・確認画面

| 観点 | 評価 | コメント |
|------|------|----------|
| 実現可能性 | ○ | 案①とセットで実装するのが自然。パース結果を一覧表示し「確定」で DB 反映。 |
| 有用性 | ○ | 誤パースの早期発見。アップロード機能の信頼性を支える。 |

**優先度**: 案①を Web 化するなら、**同一リリースまたは直後**に入れることを推奨。

---

### 6. 条件指定による一括履修登録（ユーザー案 ②の整理版）

| 観点 | 評価 | コメント |
|------|------|----------|
| 実現可能性 | ○ | `get_all_subjects` でフィルタ済みリストを取得し、各 `(day, period)` に `add_enrollment` を回す。既存の同コマ重複チェック（`crud.add_enrollment`）をそのまま使える。 |
| 有用性 | ○〜◎ | コマを逐一クリックする手間を大きく減らせる。ただし **意図の解釈**（下記）で仕様がブレやすい。 |

**ユーザー案 ②の解釈と評価**

案②は次のどちらか（または両方）と読める。

| 解釈 | 内容 | 実現可能性 | 有用性 |
|------|------|------------|--------|
| A. 一括履修登録 | 回生・分野・登録区分・曜日などで絞った科目を、該当コマにまとめて登録 | ○ | ○〜◎ |
| B. 科目マスタの自動生成 | フィルター条件だけで DB に科目を「創る」 | △〜× | △ |

現状、科目は **PDF／手動スクリプトでマスタ投入済み** なので、Web 上で意味があるのは主に **A（一括履修）** です。

**実装時に必要な UX**

- 登録前に **対象科目一覧のプレビュー**
- **同コマに複数候補**がある場合のルール（先頭のみ／ユーザー選択／スキップ）
- 失敗したコマの **理由表示**（既存の `ValueError` メッセージを活用可能）

**案②単体の優先度**: 案①・検索より後でもよいが、**「プレビュー付き一括登録」** として設計すれば有用性は高い。

## 新規提案（ユーザー案以外）

上記の表に含めたが、特に推す追加案は次のとおり。

1. **科目検索（§2）** — 低コストで体感が大きく変わる。
2. **履修 JSON のエクスポート／インポート（§4）** — データ消失リスクへの対策。
3. **PDF 取り込みプレビュー（§5）** — 案①とセットで品質を担保。
4. **履修サマリーパネル（§7）** — 時間割表だけでは見えにくい全体像の補完。

## 実装ロードマップ（推奨）

| フェーズ | 内容 | 目的 |
|----------|------|------|
| Phase 1 | 案① PDF アップロード＋ページ指定、重複対策、取り込みプレビュー | データ投入の Web 化・信頼性 |
| Phase 2 | 科目検索、フィルター永続化、履修サマリー、JSON エクスポート | 日常利用の快適化 |
| Phase 3 | 案② 条件一括履修（プレビュー・衝突処理付き） | 登録操作の効率化 |
| Phase 4 | 画像エクスポート、手動科目編集、複数案保存、単位数 | 発展機能 |

## ユーザー案の総括

| 案 | 実現可能性 | 有用性 | 推奨優先度 |
|----|------------|--------|------------|
| ① PDF アップロード＋ページ指定 | ◎ | ◎ | **最優先**（Phase 1） |
| ② 条件による自動登録 | ○（一括履修として） | ○〜◎ | **中〜高**（Phase 3。仕様を「一括履修」に固定すると実装しやすい） |

案②は「科目マスタを自動生成」ではなく **「フィルター条件に合う科目を時間割に一括で履修登録する」** と定義すると、現状の DB 設計・API と整合し、案①で投入したマスタデータの価値も活かせる。

---

## 現状実装を踏まえた制約メモ

- 履修は **同一曜日・同一時限に 1 科目まで**（`crud.add_enrollment`）。
- 科目マスタは **Web からは追加できない**（初回は CLI のみ）。
- PDF パーサは **1 大学・1 レイアウト前提**（`create_data_pdf.py`）。
- フィルタの回生は **`Subject.year <= 選択回生`**（「N 回生以上が受講可能」の意味）。

これらは新機能設計時にそのまま活かすか、明示的に仕様変更するかを決める必要がある。
